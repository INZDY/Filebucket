# Filebucket MVP Plan

## Goal
- Build Filebucket as a personal hosted file-and-note vault.
- Keep the MVP private and single-user first.
- If multiple users are created through controlled setup, keep each user isolated with one private vault.
- Use folders, Markdown notes, tags, filename/title search, trash/restore, inline image support, and backup/export.
- Use an Obsidian-inspired layout: left vault browser, center main content pane, and note outline when a Markdown note is selected.
- Keep collaboration, sharing, comments, backlinks, teams, and block-editor features out of scope unless explicitly requested.

## Architecture
- Next.js App Router is the app and backend.
- Auth.js handles login and session helpers.
- Prisma manages relational data.
- Supabase PostgreSQL is the hosted database.
- Cloudflare R2 stores media blobs.
- Browser uploads to R2 should use short-lived presigned URLs generated by Next.js.
- PostgreSQL stores users, folders, notes, tags, trash state, media metadata, and backup/export metadata.
- The vault root should be able to directly contain folders, notes, and media assets.

## Milestone 2: Vault Locations And Nested Folders
- Status: root-capable note placement, nested folder create/rename/move workflow, and same-parent folder collision checks implemented. User-verified browser smoke test completed; later media placement still pending.
- Treat the vault root as a first-class browsing and creation location.
- New vaults start with an empty root; do not require a default Inbox folder.
- Update data relationships so notes and media assets can live in the vault root or folders.
- Create folders in the vault root or inside any folder.
- Rename folders with same-parent uniqueness checks.
- Move folders between the vault root and folders while preventing cycles and destination name collisions.
- Keep folder behavior private to the current user.

## Milestone 3: Vault Browser And Content Shell
- Status: vault browser, main content pane, breadcrumbs, and note-outline shell implemented. Manual smoke test is pending; media preview/upload behavior remains in the media milestone.
- Replace the earlier folder side panel plus center list layout with:
  - left vault browser for folders, nested folders, notes, and media assets
  - center main content pane for Markdown editing and media preview
  - right note outline shown only when a Markdown note is selected
- Show the vault root as `Vault` and use UI breadcrumbs such as `Vault / Projects / Plan`.
- Make selection state linkable enough for refresh/back/forward to restore the selected location when possible.
- Show core actions together in the vault browser toolbar, such as new note, new folder, and upload media.
- Create new notes, folders, and media uploads in the currently selected vault browser location.
- Use a neutral empty state such as "Your vault is empty" without pushing a preferred first action.
- Keep authenticated user display and sign out.

## Milestone 4: Markdown Notes
- Status: note naming collisions, Markdown import, note move, rename collision rejection, and autosave failure behavior revised. Safe Markdown rendering and manual smoke testing remain pending.
- Create notes inside folders or the vault root with readable collision suffixes for default names.
- Import uploaded `.md` files as notes in the selected vault browser location, using the filename without `.md` as the default title and the file contents as the Markdown body.
- Support importing multiple `.md` files at once, creating one note per file with collision suffixes as needed.
- Keep frontmatter as note body content during Markdown import; do not parse imported frontmatter into title, tags, or metadata in MVP.
- Open new or selected notes immediately in the main content pane.
- Move notes between the vault root and folders while respecting destination name uniqueness.
- Rename note titles while rejecting explicit name collisions.
- Edit Markdown title and body.
- Autosave note changes after a 1500-2000ms pause.
- Show subtle autosave status such as "Saving..." and "Saved" near the note title or editor header.
- On autosave failure, keep the user's visible edits, show a clear error state, and allow retry by continued editing or an explicit retry action.
- Use last-write-wins for same-note edits across tabs/devices in MVP; add only a basic stale-edit warning if practical.
- Render Markdown safely, including task checkbox rendering but not raw unsafe HTML.
- Generate the right note outline from Markdown headings only; keep it navigation-only for MVP.
- Store note title and Markdown body in PostgreSQL; export writes notes out as `.md` files.

## Milestone 5: Search And Tags In The Vault Browser
- Status: needs revision. Existing search/tag code is partial and tied to the earlier layout.
- Implement whole-vault filename/title search across active folders, notes, and media assets.
- Show search results in the vault browser, not the main content pane.
- Defer full-text note-body search unless requested later.
- Add tag creation and note-tag assignment.
- Add tag rename and delete.
- Add tag filtering in the vault browser.
- Combine active search text and tag filter as an intersection over matching notes.
- Show tag-filtered notes in the vault browser, with folders or media assets only as needed context.

## Milestone 6: Trash And Restore
- Status: needs revision. Existing folder/note trash work is partial and does not cover media or the revised folder cascade rules.
- Add a Trash view for individually trashed folders, notes, and media assets.
- Soft-delete and restore folders, notes, and media assets.
- Keep folder-trash descendants hidden by the trashed folder instead of listing them as separate Trash rows unless individually trashed.
- Restore folder descendants that were hidden only by folder trash while leaving separately trashed descendants in Trash.
- Allow read-only inspection of trashed folders, note content, and media previews before restore.
- Warn when trashing referenced media, or a folder containing media referenced by active notes outside that folder.
- Defer Permanent Delete behavior until after MVP.

## Milestone 7: Media Assets
- Configure Cloudflare R2 with an S3-compatible client.
- Add a Next.js route or server action to create presigned upload URLs.
- Upload images, audio, and video directly from the browser to R2.
- Allow common image, audio, video, PDF, and TXT uploads for MVP; reject unknown or risky file types by default.
- Use an initial 100 MB per-file upload limit with a clear error when exceeded.
- Store media metadata in PostgreSQL.
- Show media assets immediately as first-class vault content in the vault browser after upload.
- Allow media assets to sit in folders or the vault root.
- Move media assets between the vault root and folders.
- Open media assets in the main content pane with image, audio, and video preview.
- Insert uploaded image references into Markdown notes.
- Keep audio/video embedding in Markdown deferred unless requested.
- Store PDFs and TXT files as media assets if uploaded, but defer in-app PDF/TXT preview to a future feature.

## Milestone 8: ZIP Export
- Export folders, notes, tags, media metadata, and media references.
- Produce one downloadable ZIP archive with folder structure, Markdown notes, media assets, and a metadata manifest.
- Treat the MVP metadata manifest as useful export metadata, not a stable ZIP re-import or restore contract yet.
- Rewrite Filebucket image media references to relative exported paths when possible.
- Keep automated backup scheduling later than MVP.

## Milestone 9: Testing And Hardening
- Add test tooling after the Prisma/Auth foundation is stable.
- Cover auth and protected routes.
- Cover folder CRUD.
- Cover note CRUD and autosave.
- Cover search and tags.
- Cover trash restore.
- Cover R2 presigned URL generation.
- Cover backup/export.
- Run `npm run lint` and `npm run build` after every implementation slice.
